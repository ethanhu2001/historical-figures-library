# Web frontend attaches to Session via a background thread and WebSocket, not a state-machine rewrite

We considered two shapes for giving `council` a browser-based frontend. The first rewrites `Session` into an explicit, resumable state machine — a `step()` method that runs until it needs the next LLM call or user input, then returns control, so each browser action is one stateless HTTP request that advances the machine by one step. The second keeps `Session.run()` exactly as it is — a single blocking call — and runs it in a background thread per active Session, with `ask_user` blocking on a `queue.Queue.get()` instead of terminal input and `on_turn` pushing each `Turn` to the browser over a WebSocket as it happens. We chose the second. ADR-0002 already committed `Session` to a suspend/resume interaction model with a synchronous, blocking shape; a state-machine rewrite would discard that shape's straight-line readability and the existing `FakeConvener`-based isolation of loop logic purely to satisfy HTTP's statelessness, when a background thread sidesteps the mismatch entirely. This also means `session.py` needs zero changes: the `AskUser`/`OnTurn`/`PickCandidates` callback seam it already exposes was already exercised by two adapters (the CLI, and hand-rolled fakes in `tests/test_session.py`) before this decision; the web backend becomes a third, using the same seam to block on a queue instead of stdin.

The Cabinet picker is handled the same way, not as a pre-flight form submitted alongside the question. `pick_candidates` is invoked by `Session.run()` only after `needs_debate()` returns `True` (mirroring how a Clarifying Question can only occur mid-debate), so treating "choose a Cabinet" as a suspend/resume WebSocket event — the backend sends a `choose_cabinet` message with the full Library only once it knows a debate is actually happening — avoids showing the browser a picker for a question that turns out trivial, and reuses the identical mechanism as answering a Clarifying Question rather than inventing a second one.

We scoped the backend to exactly one active Session at a time, in memory, rejecting a concurrent `POST /ask` with an error rather than building a session registry keyed by id or adding cooperative cancellation to replace a running Session. This matches how the tool is actually used (one person, one question at a time) and defers session-registry complexity to if/when multi-session use is a real requirement, not a hypothetical one. The new adapter lives at `src/council/web.py`, alongside `cli.py`, importing `Convener`/`Session`/`Figure` directly with no new package boundary; a new `council serve` command runs it via `uvicorn` and also serves the built React+TypeScript+Vite frontend (in a new top-level `frontend/` directory, styled with Tailwind) through `StaticFiles`, so everyday use is one process and one command. The Vite dev server with an API proxy is only used during active frontend development, not for regular use.
